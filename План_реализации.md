# План реализации проекта «Quest Guide for Tourists»

## 1) Краткий анализ ТЗ

### Что уже хорошо сформулировано
- Понятная цель: self-guided туризм в формате квеста.
- Полный пользовательский сценарий: выбор маршрута -> навигация -> задания -> награды -> история.
- Нефункциональные требования и high-level архитектура уже заданы.

### Что нужно уточнить до начала активной разработки
- Режим офлайн: только кеш карты/контента или полноценное прохождение без сети.
- Модель проверки ответов: строгая/нестрогая, ручная проверка фото-заданий, античит.
- Гео-правила: радиус активации точки, допуск по точности GPS.
- Аудиогид: обязателен в MVP или переносится на v2.
- Рейтинг: глобальный, по городу, по периоду времени.
- Роли в админке: только администратор или редактор + модератор.
- Правовые требования: политика конфиденциальности, обработка геоданных, age/consent.

## 2) Рекомендуемый стек (под текущий репозиторий)

С учетом того, что проект уже создан на Flutter:
- Мобильный клиент: Flutter (Android), state management `Riverpod`.
- Бэкенд: Firebase (`Auth`, `Firestore`, `Cloud Functions`, `Storage`, `FCM`).
- Карты: Google Maps SDK + гео-утилиты (`geolocator`, `geofencing`/ручной proximity check).
- Аналитика: Firebase Analytics + Crashlytics.
- Локальное хранение: `Hive` или `Isar` для кеша маршрутов/прогресса.
- Админ-панель: Flutter Web или React + Firebase SDK (вторая команда может делать параллельно).

Почему так:
- Быстрый time-to-market для MVP.
- Минимум DevOps на старте.
- Хорошая поддержка realtime и медиа.

## 3) Предлагаемая архитектура

### Мобильное приложение (Clean-ish feature architecture)
- `features/auth`
- `features/quests` (список городов, маршрутов, карточки)
- `features/map_nav` (карта, трекинг, proximity)
- `features/challenges` (типы заданий, проверка)
- `features/rewards` (очки, бейджи, рейтинг)
- `features/history` (прошлые прохождения, статистика)
- `features/profile` (язык, настройки)
- `core` (theme, routing, network, error handling, localization)

### Серверный слой
- Firestore как основная БД.
- Cloud Functions:
  - проверка сложных ответов;
  - начисление баллов/бонусов;
  - выдача достижений;
  - поддержка лидерборда.

### Основные сущности данных
- `users`
- `cities`
- `quests`
- `quest_points`
- `tasks`
- `attempts`
- `progress`
- `achievements`
- `leaderboard_snapshots`

## 4) MVP (что обязательно в первую версию)

- Email/Google/Apple login.
- Выбор города и маршрута.
- Карта + текущее местоположение + переход к следующей точке.
- 3 типа заданий: тест, текстовый ответ, загадка.
- Начисление очков за правильные ответы.
- Сохранение и восстановление прогресса.
- История прохождения.
- RU/KZ локализация.
- Базовая админка: создание маршрута, точек и заданий, загрузка фото.

Не включать в MVP:
- Полный аудиогид.
- Сложные фото-задания с ML-проверкой.
- Социальные механики (друзья/команды).

## 5) Пошаговый план реализации (10 недель, 5 спринтов)

### Спринт 0 (Неделя 1): Discovery + проектирование
- Финализировать open questions из раздела 1.
- Описать user flows и acceptance criteria.
- Утвердить data model Firestore и правила безопасности.
- Подготовить wireframes ключевых экранов.
- Настроить CI (lint/test/build), Crashlytics, Analytics.

Результат:
- Архитектурный baseline + backlog на MVP.

### Спринт 1 (Недели 2-3): Базовый каркас приложения
- Инициализация архитектуры по фичам.
- Авторизация (email/google/apple), профиль пользователя.
- Локализация RU/KZ, базовая дизайн-система.
- Экран города/маршрутов (пока на seed-данных).

Результат:
- Пользователь может войти и выбрать маршрут.

### Спринт 2 (Недели 4-5): Карта и навигация
- Интеграция Google Maps.
- Разрешения геолокации + обработка отказов.
- Отображение точек маршрута и текущей позиции.
- Механика активации точки по радиусу.

Результат:
- Рабочий навигационный контур квеста.

### Спринт 3 (Недели 6-7): Квест-механика и награды
- Экран задания + проверка ответов.
- Начисление очков и бонуса за скорость.
- Выдача достижений.
- Сохранение прогресса и история прохождений.

Результат:
- End-to-end прохождение квеста от старта до финиша.

### Спринт 4 (Недели 8-9): Админка + контент
- Базовая админ-панель (CRUD маршрутов/точек/заданий).
- Загрузка медиа в Storage.
- Валидация контента и preview маршрута.

Результат:
- Контент можно заводить без разработчика.

### Спринт 5 (Неделя 10): Stabilization + релиз
- Полный QA: функциональный, UX, регресс.
- Оптимизация производительности (цель <=2 сек на ключевых экранах).
- Подготовка App Store / Google Play материалов.
- Soft launch на ограниченную аудиторию.

Результат:
- Готовность к production запуску.

## 6) План тестирования

- Unit: проверка бизнес-логики (очки, бейджи, валидация ответов).
- Widget: ключевые экраны (маршрут, задание, результат).
- Integration: сценарий полного прохождения квеста.
- Field testing: реальный маршрут в городе с плохим GPS/интернетом.
- Security: проверка Firestore rules и авторизации.

## 7) Ключевые риски и меры

- Неточный GPS в плотной застройке.
  - Мера: плавающий радиус + повторная проверка координат + ручное подтверждение на спорных точках.
- Большая вариативность текстовых ответов.
  - Мера: нормализация текста + словарь синонимов + fallback на приблизительное совпадение.
- Слабый интернет у туристов.
  - Мера: локальный кеш маршрута и прогресса, retry-очередь.
- Задержки публикации в сторы.
  - Мера: заранее подготовить legal-страницы и метаданные релиза.

## 8) Критерии готовности MVP (Definition of Done)

- Пользователь проходит минимум 1 квест от начала до конца без критических ошибок.
- Прогресс не теряется при закрытии приложения/перезапуске.
- Основные экраны открываются не дольше 2 секунд на целевых устройствах.
- Есть минимум 2 города и 3 маршрута с валидным контентом.
- Crash-free sessions >= 99% на тестовой группе.
